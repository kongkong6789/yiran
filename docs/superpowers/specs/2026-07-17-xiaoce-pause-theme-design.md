# 小策bot暂停生成、全站黑白主题与会话 Skill 打包设计

## 背景

当前正式的 AI 对话入口已经迁入 `/collab`，以通讯录中的“小策bot”单聊形式工作。旧的暂停生成和会话打包 Skill 实现仍停留在 `codex/chat-pause-skill-packaging` 分支，并且只服务于旧 `/agent` 会话模型；已经上线到 `main` 的苹果黑白主题同样只作用于旧页面。

本次采用最小迁移：保留协作页面和小策bot现有消息、WebSocket、轮询及知识问答链路，不整体合并旧分支，只迁移真实取消、已确认的黑白主题，以及经过安全校验的会话打包 Skill 能力。

## 目标

1. 小策bot生成回答时，右下角发送按钮切换为方形暂停按钮。
2. 点击暂停会真实终止本轮后端执行，保留用户消息，不保存未完成回答，并且只显示一次“已暂停本次生成”。
3. 小策bot会话头部提供纯白与纯黑主题切换，并通过全局主题状态同步改变整个网站；默认白色并记住用户选择。
4. 用户用明确自然语言指令把当前小策bot会话提炼为私有 Skill，自动上传并启用。
5. 同一会话重复打包更新同一 Skill，不产生重复资产；打包过程也可以暂停。
6. Logo、用户头像、小策bot头像和风险状态等有业务含义的颜色保持原样；页面背景、表面、文字、边框和控件使用全局黑白变量。
7. 尽量沿用同事现有代码结构，不重写协作聊天。

## 不在本次范围

- 不在普通同事私聊或群聊头部重复增加主题按钮；全站仍会跟随已选择的主题。
- 不改变 `/collab` 的会话结构、实时同步协议或风险监控逻辑。
- 不整体合并旧功能分支；只迁移暂停与会话打包所需的独立服务和数据字段。
- 不自动把会话 Skill 发布给团队或共享仓库。
- 不把原始聊天全文或模型凭空生成的可执行脚本写入 Skill 包。
- 不保留部分生成内容，也不提供从中断位置继续生成。
- 不改变 `/agent` 路由和页面。
- 不负责内网服务器部署；代码审核通过后再按现有上线流程更新服务器。

## 方案选择

采用持久化运行记录、协作式取消，以及直接适配 `CollabRoom` 的服务端会话打包。

仅使用浏览器 `AbortController` 虽然改动少，但后台仍会继续调用 LLM、Skill 和 MCP，不满足真实暂停。把协作消息复制到旧 `ChatSession` 会形成两套历史并产生同步问题；由浏览器直接生成 ZIP 也无法可靠完成脱敏、校验和事务回滚，因此都不采用。

## 后端设计

### 小策bot运行记录

在 `collab` 应用新增独立运行模型，记录：

- 前端生成的 UUID；
- 房间、发起用户和触发消息；
- `running`、`cancelled`、`completed`、`failed` 状态；
- 唯一的暂停占位消息；
- 取消、完成、创建和更新时间及错误信息。

运行记录属于协作会话，不复用旧 `ChatRun` 对 `ChatSession` 的强关联，避免把两套会话模型硬绑在一起。

### 发送与执行流程

1. 小策bot单聊发送消息时，前端生成 `run_id` 并随现有消息请求提交。
2. 后端保存用户消息和运行记录后，继续用现有后台线程执行小策bot回答；发送接口仍然快速返回。
3. 后台线程把统一 `cancel_check` 传入现有 `run_chat`，并在 RAG、图谱、MCP、Skill、图片及 LLM 阶段边界检查取消状态。
4. LLM 使用旧暂停实现已经验证的流式读取方式，在数据块之间检查取消并关闭连接；Skill 子进程和 MCP 调用沿用旧分支的协作式取消方式。
5. 后台线程先判断本轮是否为明确的会话打包指令：普通消息继续走 `run_chat`，打包指令改走独立的会话 Skill 服务；两条路径使用同一个 `cancel_check`。
6. 回答或打包结果生成后，在事务中锁定运行记录：只有仍为 `running` 才创建小策bot消息并标记 `completed`；若已经取消则丢弃结果和未提交资产。
7. 运行状态变化继续通过现有 WebSocket 推送，并由轮询兜底。房间详情返回当前用户在该小策bot房间里的活动运行，刷新或切回页面后仍能恢复暂停按钮。

### 取消接口

新增小策bot运行取消接口，要求：

- 只能取消当前登录用户在当前房间发起的运行；
- `running` 转为 `cancelled`，并以事务创建唯一的“已暂停本次生成”小策bot消息；
- 重复取消返回相同结果，不重复创建消息；
- 已 `completed` 的运行返回冲突，避免把已完成回答伪装成暂停；
- 不存在或不属于当前用户的运行统一返回未找到；
- 取消结果和占位消息通过现有实时通道推送。
- PostgreSQL 行锁查询只锁定 `XiaoceRun` 主表以及非空的房间关联，不在 `SELECT FOR UPDATE` 中连接可为空的 `cancel_message`；幂等取消需要旧消息时再单独读取，避免 PostgreSQL 的 nullable outer join 锁定错误。

### 会话打包 Skill

新增独立服务适配 `CollabRoom` 消息，不把生成逻辑堆入视图：

1. 只有消息同时表达“当前对话/聊天记录”“Skill/技能”以及“打包/生成/上传”等明确语义时才触发；普通 Skill 咨询仍走正常问答。
2. 按时间读取当前小策bot单聊的用户与助手消息，排除本次打包指令、系统消息、已删除/撤回消息和“已暂停本次生成”占位。
3. 至少存在一轮完整用户问题与小策bot回答才允许打包。
4. 在送入模型前识别并替换 API Key、Bearer Token、Cookie、密码和常见密钥格式；限制总字符数并优先保留任务目标、关键决策和最终结果。
5. 模型只返回严格 JSON：名称、描述、命令式执行说明和工作流摘要。执行说明必须包含目标、输入、步骤、输出、验证和失败处理。
6. 生成 ZIP 只包含 `SKILL.md` 与 `references/workflow-summary.md`；原始聊天全文不写入包，模型不得凭空加入 `scripts/`。
7. 使用现有 Skill parser 再次校验目录、frontmatter 和内容后，才在事务中保存资产并为当前用户启用。
8. Skill ID 由会话首个任务型问题和房间短 ID 稳定生成；同一房间重复打包更新同一 Skill。

### 私有 Skill 数据规则

为 `SkillAsset` 增加 `private`/`shared` 可见性，现有资产迁移为 `shared`，保持团队仓库行为不变。会话打包只创建 `private` 资产，并立即生成当前用户的 `UserSkill`：

- 共享列表、共享查找和全员自动同步只处理 `shared`；
- 私有资产只能被上传者读取、更新、删除和启用；
- 上传或数据库任一步失败都回滚，不暴露半成品；
- 取消发生在资产提交前时立即终止；提交事务开始后由运行锁决定最终状态，防止“界面显示已暂停但 Skill 已创建”。

### 消息元数据

为协作消息增加内部 `meta`，用于持久化 `run_id`、`cancelled` 和 `created_skill`。WebSocket、轮询和历史接口返回同一元数据，使刷新后仍能显示 Skill 结果和运行状态；普通消息的 `meta` 默认为空，不改变现有展示。

## 前端设计

### 运行交互

- 只有当前房间是小策bot单聊且存在活动运行时，发送按钮才变为方形暂停按钮。
- `running` 显示“暂停生成”；点击后进入 `cancelling`，显示“正在暂停”并阻止重复点击。
- 取消成功后，由服务端实时消息显示“已暂停本次生成”，按钮恢复为发送。
- 取消失败时恢复可点击暂停按钮并保留 `run_id`，允许重试。
- 打包成功消息显示 Skill 名称、ID、“仅自己可见”和“已自动上传并启用”，并触发 Skill 选择器刷新。
- 打包失败显示可重试的具体失败原因，不创建资产，也不回退成要求用户手工制作 ZIP。
- 普通协作会话继续使用现有发送按钮和发送逻辑。

### 全站主题

- 小策bot会话头部仍显示白色/黑色两个按钮，但按钮直接调用项目现有的 `ThemeModeContext`，不再维护第二份局部聊天主题状态。
- 统一使用现有全站存储键 `lc-theme-mode`；切换后由根节点的 `data-theme`、Ant Design `ConfigProvider` 和 CSS 变量同步更新，刷新及访问其他路由时继续保持。
- 主题开关只在小策bot会话头部显示。
- 白色主题以纯白 `#fff` 为画布和表面、纯黑 `#000` 为文字及主控件；黑色主题反向使用纯黑画布和表面、纯白文字及主控件。
- 次要文字、边框、悬停态和阴影仅使用当前前景色的透明度，不增加独立中性灰色主题或灰色表面。
- 移除小策bot局部黑屏覆盖，让聊天区、左侧会话列表、右侧监控、顶部导航、弹窗和其他路由共用同一组全局变量；保留各区域原有结构与密度。
- Logo、头像不使用滤镜或强制改色；成功、警告、风险和在线状态等语义色继续保留，避免丢失业务含义。

## 一致性与竞态

- 同一用户在同一小策bot房间只允许一个活动运行；生成中再次发送会被前端阻止，后端也做冲突校验。
- 完成和取消通过数据库行锁决定唯一最终状态，保证不会同时出现正式回答和暂停占位。
- 会话 Skill 的资产保存与个人启用在同一事务中；失败或取消不会留下半成品。
- 同一会话的稳定 Skill ID 保证重复打包更新原资产，同时保留其他会话生成的 Skill。
- 切换房间不会自动取消；返回小策bot房间时从服务端恢复活动运行状态。
- 组件卸载或浏览器断开不会伪造暂停，后台运行仍由持久化状态管理。

## 测试与精简验证

后端聚焦测试：

- 运行创建、所有权、活动运行冲突和状态转换；
- 幂等取消、暂停消息唯一、完成后不可取消；
- 取消后不保存正式回答，以及完成/取消竞态；
- `run_chat`、LLM、MCP 和 Skill 在关键阶段响应取消；
- 小策bot发送接口返回运行状态，房间详情可恢复活动运行；
- 打包意图正反例、有效历史筛选、暂停消息排除和凭据脱敏；
- 合法 ZIP 结构、稳定 Skill ID、重复更新、私有资产隔离和事务回滚；
- 打包过程中取消时不生成资产、不保存成功消息。

前端聚焦测试：

- 仅小策bot生成时显示暂停按钮；
- 取消中防止重复点击，失败后允许重试；
- 收到完成或暂停状态后恢复发送按钮；
- 主题默认白色、切换黑色并持久化；
- 主题选择器不会出现在其他协作会话，但所选主题会覆盖全站；
- 黑色模式下导航、会话列表、消息正文、Markdown、输入框、监控面板和弹窗均保持可读；
- PostgreSQL 下取消接口不会因可空外连接行锁返回 500；
- 打包成功消息展示创建结果并刷新 Skill 选择器；失败或暂停不显示成功标签。

验证流程保持精简：先跑相关后端测试和前端单测，再各跑一次前端构建与 Django 系统检查，最后只在浏览器验证小策bot的发送、暂停、刷新恢复、黑白切换、成功打包及重复更新。

## 验收标准

- 小策bot思考期间可以真实暂停，暂停后不再出现本轮正式回答。
- 用户消息保留，暂停占位只出现一次，随后可以继续提问。
- 明确打包指令会生成一个仅自己可见、自动上传并启用的 Skill；重复执行更新同一个 Skill。
- Skill 包不包含原始聊天全文、凭据或模型凭空生成的脚本；失败和暂停均不留下半成品。
- 黑白主题切换会改变整个网站，刷新及切换路由后仍保持选择。
- 黑色模式中的消息正文和控件达到清晰可读的对比度，不再出现白卡白字或局部黑屏。
- 小策bot Logo、品牌 Logo、用户头像及风险/在线状态颜色保持原样。
- 现有协作实时消息、轮询兜底、附件发送和风险监控测试无回归。
