# 全站暗色模式兼容设计

日期：2026-07-20  
状态：已批准  
分支：`codex/dark-mode-compatibility`

## 1. 背景

平台已经支持浅色/暗色切换，但当前暗色模式主要依赖全局黑白覆盖。部分页面和可视化组件仍使用硬编码浅色背景、深色文字或只适用于浅色主题的 Canvas/SVG 配色，因此出现以下问题：

- 页面主体已变黑，但卡片、输入框、表格、代码区或画布仍为白色，形成突兀的“白底孤岛”。
- 纯黑背景上的多个区域缺乏层级，边界、悬浮态和禁用态难以分辨。
- 图谱、经营闭环和关系画布的背景、标签底色、tooltip 与网格仍使用浅色常量。
- 局部样式用大量 `!important` 强制黑白，掩盖了组件语义且增加后续维护风险。
- 一些文字、占位符、状态标签和边框在暗色背景下对比不足。

本次工作是对现有纯黑主题做系统兼容，不新增或更换主题。

## 2. 目标

1. 保留现有纯黑、克制、企业级的视觉方向。
2. 让所有正式路由及其子模块在暗色模式下自然一致，无突兀浅色区域。
3. 用统一语义变量表达页面、容器、浮层、边框、文字和交互状态。
4. 仅在具有业务含义的位置保留低饱和功能色。
5. 同时保护浅色模式与现有交互功能，不引入布局回归。
6. 建立可重复的静态检查、组件测试和页面截图验收方法，防止新代码重新引入暗色缺陷。

## 3. 非目标

- 不重新设计品牌、导航结构或信息架构。
- 不把纯黑主题替换为石墨、深蓝或其他新主题。
- 不改变业务流程、接口契约、权限与数据逻辑。
- 不为了视觉统一而移除图谱节点、状态、品牌连接器等原本具有识别意义的颜色。
- 不在本次工作中重写所有页面组件或迁移 UI 框架。

## 4. 已批准的视觉方向

采用“纯黑主体 + 克制功能色”。

### 4.1 表面层级

暗色模式不使用大面积灰色替代纯黑，而是在纯黑主体上建立最小但可见的层级：

- 页面底层：纯黑，用于应用背景和大面积画布外区域。
- 一级容器：近黑，用于侧栏、主卡片、表格与常驻面板。
- 二级容器：略亮的深灰，用于输入框、列表项、内嵌卡片和选中内容。
- 浮层：比承载容器更清晰的深灰，用于下拉、Popover、Modal、Tooltip 和悬浮操作区。
- 边框：默认边框、强调边框与焦点边框分别使用不同强度，避免区域粘连。

这些颜色必须通过 CSS 变量和组件令牌引用，不在页面中重复写死。

### 4.2 文字层级

- 主文字：接近白色，避免纯白大面积造成眩光。
- 次级文字：中高亮灰，仍满足常用字号的可读性。
- 辅助文字：用于时间、说明和占位符，但不可低到无法辨认。
- 禁用文字：同时配合透明度、图标或结构状态表达，不只靠低对比度。

### 4.3 功能色

页面结构保持黑白灰；以下场景允许使用低饱和功能色：

- 成功/在线/已完成：绿色。
- 警告/待确认/部分完成：琥珀色。
- 失败/风险/删除：红色。
- 信息/链接/处理中：蓝色。
- 图谱节点、经营闭环、连接器品牌：保留分类识别色，但降低亮度冲击并保证标签文字可读。

状态必须同时包含文字、图标、形状或边框之一，不能只靠颜色区分。

## 5. 主题架构

### 5.1 单一主题来源

`ThemeModeContext` 继续作为浅色/暗色状态来源，并同步到 `html[data-theme]`。所有页面、Ant Design 组件和可视化组件均从这个状态派生样式，避免页面单独维护第二套互不一致的主题状态。

### 5.2 语义变量

在全局样式中整理并补齐语义变量，至少覆盖：

- 页面与容器背景。
- 浮层、输入框、悬浮、选中和禁用背景。
- 主文字、次级文字、弱文字、反色文字。
- 默认、弱、强调和焦点边框。
- 阴影、遮罩、滚动条与网格。
- 成功、警告、错误、信息及其弱背景。
- 聊天中的本人气泡、AI 气泡、系统提示和代码块。
- 图谱画布、网格、边、标签底色和 tooltip。

已有 `--lc-*` 变量继续保留兼容，并逐步映射到更完整的语义层。禁止新增只针对某一个页面的无语义黑白常量。

### 5.3 Ant Design 令牌

`ConfigProvider` 的暗色令牌与 CSS 变量保持一致，覆盖 Layout、Card、Table、Tabs、Input、Select、Button、Modal、Drawer、Popover、Tooltip、Dropdown、Pagination、Segmented、Alert、Tag 和 Empty 等常用组件。

组件令牌负责框架级默认值；页面 CSS 只负责布局与业务态，不重复覆盖基础背景和文字。

### 5.4 Canvas、SVG 与第三方可视化

Canvas、SVG、React Flow、ECharts/Force Graph 等无法自动继承 CSS 文字颜色的区域，需要显式接收当前主题并使用主题调色板：

- 背景与网格切换为暗色。
- 节点填充、外框、标签与边在暗色下重新计算可见性。
- label background、HTML tooltip、节点详情浮层使用暗色语义变量或主题调色板。
- 分类色保持含义，但使用暗色专用 soft 色或带透明度的深色底。
- 主题切换后触发必要的重绘，不要求刷新页面。

## 6. 页面与模块覆盖

### 6.1 应用外壳与通用组件

- 顶部导航、侧栏、面包屑、通知、用户菜单。
- 全局 Modal、Drawer、Dropdown、Select 下拉、Tooltip、Popover、Message、Notification。
- 表格固定列、分页、筛选器、表单校验、空状态、骨架屏、加载状态。
- Markdown、代码块、引用、链接、附件、滚动条和焦点态。

### 6.2 Agent 对话

- 历史对话侧栏、对话主区和顶部状态栏形成清晰但克制的黑色层级。
- 用户气泡保持反色识别，但使用柔和白而非刺眼纯白；AI 气泡、系统提示、处理过程、错误态和引用区全部适配暗色。
- 输入器、模型选择、Skill/连接器入口、附件和发送按钮在默认、悬浮、聚焦、禁用状态下均可辨认。
- Markdown 表格、代码、列表和链接不出现浅色底或深色不可读文字。

### 6.3 团队协作

- 会话列表、聊天区、智能纪要、圆桌、成员弹窗、邀请提醒、风险与监控面板统一适配。
- 保留成员头像和状态色，但移除只适用于浅色的淡紫、淡蓝、淡红背景。
- 小策处理过程在暗色下保持步骤层级、成功/失败状态和折叠区域可读。

### 6.4 Agent 记忆

- 页面标题、说明、Tabs、筛选、表格、表单、空状态和 Context 调试结果形成完整容器层级。
- 调试 `<pre>`、JSON、命令输出和长文本不再回落到浅色默认背景。
- 内容为空时仍保留结构和适当的视觉重心，避免整页成为无层次纯黑空洞。

### 6.5 本体图谱

- 主画布、网格、边、节点标签、tooltip 和画布状态条均使用暗色调色板。
- 顶部工具栏、筛选、统计条、提示条、添加对象、选中对象、AI 抽取和关系列表统一为暗色容器。
- 图谱分类色保持区分，标签和边在缩放、选中、悬浮状态下仍可读。
- 关系表格、分页和编辑弹窗不得出现白底孤岛。

### 6.6 经营与闭环画布

- Commerce、Loops、Fusion 以及各类 Stock/Flow/Loop Canvas 的 SVG、React Flow 背景、节点 soft 色、边标签和 tooltip 全部具备暗色分支。
- 上卷、因果、负向关系和数据接入状态保留语义色。
- 画布与其工具栏、侧栏、图例在同一视觉层级系统中。

### 6.7 其他正式路由

覆盖 Home、Work、Knowledge、Skills、Connectors、SmartTable、DataLake、Agents、Accounts、Audit、SectionHub 和 Login 等正式页面，以及它们加载的共用组件。

对尚未被截图暴露的问题，使用路由级巡检和静态颜色审计发现并修复，而不是只修当前三张截图。

## 7. CSS 整理策略

当前全局样式末尾存在强制黑白的“最终层”。本次不会简单继续叠加覆盖，而是：

1. 识别仍有必要的全局兜底规则。
2. 把通用背景、文字和边框迁移到语义变量或 Ant Design 令牌。
3. 缩小 `!important` 的选择器范围，仅保留第三方组件确实需要的兼容覆盖。
4. 为 Canvas/SVG 等不可继承区域提供显式暗色调色板。
5. 对浅色专用 soft 背景增加对应暗色值，不直接把所有状态压成纯黑。

## 8. 交互与状态

- 主题切换立即生效，页面无需刷新。
- Hover、Active、Selected、Focus、Disabled、Loading、Error 和 Empty 状态都必须在暗色下可辨认。
- 焦点环在纯黑和近黑表面上都有足够对比度。
- 遮罩与浮层层级清晰，Modal、Drawer 和下拉层不会与底层融合。
- 不改变现有键盘、鼠标、拖拽、缩放、折叠和分页行为。

## 9. 可访问性与视觉验收

- 普通正文目标对比度不低于 4.5:1。
- 大字号文字、图标和关键控件边界目标不低于 3:1。
- 状态不只用颜色传达。
- 纯白仅用于小面积高优先级内容，长文本采用柔和白。
- 默认、悬浮、选中和禁用状态可以在实际显示器亮度下辨认。
- 1440px 桌面宽度为主要验收尺寸，同时检查常见中等宽度，确保没有因样式调整造成溢出。

## 10. 测试与验证

### 10.1 自动化测试

- 扩展主题测试，验证全局模式切换和持久化。
- 为暗色调色板/语义变量映射增加针对性测试。
- 为关键 Canvas/图谱组件增加主题分支或主题属性测试。
- 保留并运行现有前端测试套件。

### 10.2 静态审计

- 扫描页面、组件、CSS、Canvas 和内嵌 tooltip 中的硬编码白底、浅灰底和浅色专用深文字。
- 允许品牌色、数据分类色和明确的浅色主题常量，但必须有暗色对应逻辑。
- 检查新样式是否继续扩大无边界的全局 `!important` 覆盖。

### 10.3 构建与页面巡检

- 运行 TypeScript/前端测试与生产构建。
- 启动本地服务，在暗色与浅色模式下逐个访问所有正式路由。
- 对关键页面截取对比图：Agent 对话、团队协作、Agent 记忆、本体图谱、经营闭环、工作台、知识库、Skill、连接器、智能表格和管理页面。
- 检查控制台错误、网络错误、布局溢出、白底孤岛、低对比文字和不可见边界。

## 11. 实施顺序

1. 增加或修正测试，锁定主题状态、语义调色板和关键组件行为。
2. 整理全局语义变量和 Ant Design 暗色令牌。
3. 修复应用外壳与通用浮层/表单/表格组件。
4. 修复截图暴露的 Agent 对话、Agent 记忆和本体图谱。
5. 修复经营闭环、关系画布及其他 Canvas/SVG 组件。
6. 按正式路由逐页巡检并修复剩余问题。
7. 完成测试、生产构建、静态审计和双主题截图验收。

## 12. 风险与控制

- 风险：全局 CSS 调整影响浅色模式。  
  控制：变量分主题声明，关键页面进行双主题截图比对。

- 风险：Canvas/SVG 的主题切换不触发重绘。  
  控制：将主题作为依赖或属性传入，并为切换路径增加测试。

- 风险：过度追求黑白统一导致状态难辨。  
  控制：保留克制功能色，并同时使用文字、图标或边框表达状态。

- 风险：一次性覆盖全站导致回归面较大。  
  控制：按“全局基础 → 关键页面 → 可视化 → 其余路由”分阶段验证，每阶段保持可构建、可测试。

## 13. 完成定义

满足以下条件后视为完成：

- 所有正式路由在暗色模式下完成巡检并无明显白底孤岛或不可读内容。
- 截图中的 Agent 对话、Agent 记忆和本体图谱问题全部消失。
- 经营闭环与各类图谱画布具备暗色背景、标签和 tooltip。
- 浅色模式核心页面无明显视觉回归。
- 主题相关测试、现有前端测试和生产构建通过。
- 静态颜色审计中的剩余硬编码均有明确语义或暗色分支。
- 最终提供本地预览地址和关键页面验收说明。
