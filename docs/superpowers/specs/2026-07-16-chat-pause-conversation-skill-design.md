# 对话暂停与会话生成 Skill 设计

## 目标

在“良策 AI”对话页增加两项平台原生能力：

1. 用户发送消息后，可以立即暂停本轮执行。暂停必须停止仍在进行的 LLM 输出和 Skill 脚本，并阻止后续 RAG、MCP、Skill 或回答落库步骤继续运行。用户消息保留，会话中显示“已暂停本次生成”，之后可继续发消息或重新生成。
2. 用户在完成一轮多轮任务后，说“把这次对话打包成 Skill 并上传平台”等自然语言指令，平台读取当前会话的完整历史，提炼可复用工作流，生成合法 Skill zip，保存为当前用户的私有 Skill 并自动启用。用户无需下载、复制文件或手动上传。

## 范围与产品规则

- 暂停是后端可观察的真实取消，不是单纯隐藏加载状态或丢弃浏览器响应。
- 暂停保留本轮用户消息，不保存未完成的模型内容。
- 取消接口必须幂等；重复点击只产生一条“已暂停本次生成”消息。
- 生成 Skill 默认只对当前登录用户可见并自动启用。
- 私有 Skill 不进入团队共享仓库；用户以后可在 Skill 页面主动发布为共享资产。
- 打包范围是当前会话的全部消息，排除本次打包指令、暂停占位消息和明显无关闲聊。
- 生成包不保存原始聊天全文，不复制 API Key、Token、Cookie、密码或其他凭据，只保存脱敏、抽象后的工作流与参考摘要。
- 同一会话重复执行打包时更新同一个会话 Skill，避免重复堆积；Skill ID 由会话首个任务型用户消息生成的稳定 slug 与会话短 ID 组成，不依赖模型每次可能变化的命名。
- 只有明确同时包含“会话/这次对话/聊天记录”“Skill/技能”和“打包/生成/上传”等语义的命令才触发自动打包，普通的 Skill 咨询继续走正常对话。

## 方案比较

### 方案 A：只使用浏览器 AbortController

前端可立即结束等待，但 Django、MCP、脚本和上游 LLM 仍可能继续运行并保存回答。这不满足真实暂停，拒绝采用。

### 方案 B：持久化运行记录与协作式取消（采用）

每次发送由前端生成 `run_id`，后端创建 `ChatRun`。暂停接口立即把运行标记为取消并写入唯一占位消息；执行链在阶段边界检查状态，Skill 子进程使用可终止的 `Popen`，LLM 使用流式连接并在每个数据块检查取消状态。该方案只增加少量模型、API 与取消检查，不引入新的队列基础设施。

### 方案 C：Celery/Redis 异步任务与 SSE

任务队列最适合多实例和长任务，但需要新增 worker、broker、部署与故障恢复策略，并重写现有同步对话协议。本期不采用；未来出现多实例水平扩展或离线长任务时，可以让 `ChatRun` 成为迁移基础。

## 架构

### 运行与取消

新增 `ChatRun`，字段包括：

- `id`：前端创建的 UUID。
- `user`、`session`：所有权与所属会话。
- `status`：`running`、`cancelled`、`completed`、`failed`。
- `cancel_message`：关联唯一的暂停占位消息，用数据库关系保证幂等。
- `cancelled_at`、`finished_at`、`error`、`created_at`、`updated_at`。

发送流程：

1. 前端生成 `run_id`，把用户消息立即加入页面并提交 `/api/agent/chat/`。
2. 后端创建会话（如需要）、保存用户消息并创建 `ChatRun(running)`。
3. `run_chat` 在解析 Skill、执行脚本、RAG、图谱、MCP、图片处理和 LLM 前后调用同一个 `cancel_check`。
4. 普通完成时保存助手回答并把运行标为 `completed`；异常时标为 `failed`。
5. 暂停按钮调用 `POST /api/agent/runs/{run_id}/cancel/`。接口校验当前用户所有权，把运行改为 `cancelled`，并以 `run_id` 为幂等键写入一条助手占位消息。
6. 原请求检测到取消后停止处理，返回 `cancelled: true`，不再写第二条回答。

LLM 取消通过 OpenAI 兼容接口的 `stream: true` 实现。后端逐行读取 SSE，在数据块之间检查 `cancel_check`；取消时关闭响应连接并抛出受控的 `ChatRunCancelled`。不带取消检查的其他调用保留现有非流式路径，控制改动范围。

Skill 脚本从 `subprocess.run` 改为 `Popen` 轮询。取消时终止整个进程组，超时先温和终止、必要时强制结束，确保脚本不会在页面显示暂停后继续写文件或访问外部系统。

MCP 客户端接收同一 `cancel_check`，在 initialize、tools/list、每个 tools/call 之前和之后检查取消状态，并在读响应时关闭本地 HTTP 连接。已经发送给第三方 MCP 服务的请求无法被平台“撤回”，服务端可能仍会完成该次只读调用；但平台会立即放弃其结果，不再发起下一次 MCP 调用，也不会把返回内容交给 LLM。该边界会在代码注释与测试中明确。

### 会话生成 Skill

新增独立服务 `conversation_skill.py`，不把生成逻辑堆进 API 视图。服务负责：

1. 判断消息是否是明确的会话打包指令。
2. 从数据库按顺序读取当前会话全部消息，移除当前打包指令、暂停消息和空内容。
3. 对输入做凭据模式清理，并限制总字符数；超长会话保留任务目标、关键决策和最终结果，优先删除重复闲聊。
4. 调用当前用户已配置的 LLM，要求返回严格 JSON：`name`、`description`、`instructions`、`workflow_summary`。
5. 校验并规范化结果：名称、描述、命令式正文、必需章节、无危险 frontmatter、无凭据片段。
6. 生成内存 zip：根目录包含 `SKILL.md` 与 `references/workflow-summary.md`。只有会话中存在已验证、可复用的确定性脚本时才允许加入 `scripts/`；模型不得凭空生成可执行脚本。
7. 使用现有 Skill repository 保存包、创建个人 `UserSkill` 并启用。
8. 返回结构化 `created_skill`，正常对话消息显示名称、`@skill-id`、私有状态和“已自动上传并启用”。

如果排除打包指令后没有至少一轮完整的用户问题与助手结果，服务返回“当前会话内容不足以提炼 Skill”，且不创建资产。

生成的 `SKILL.md` 只使用 `name` 和 `description` 两个 YAML frontmatter 字段。正文使用命令式表达，并至少包含：目标、适用边界、输入、执行步骤、输出、校验与失败处理。`references/workflow-summary.md` 保存脱敏后的目标、关键决策和可复用知识，不保存逐字聊天记录。

### 私有与共享 Skill

为 `SkillAsset` 增加 `visibility`：`private` 或 `shared`。

- 现有资产迁移为 `shared`，保持当前团队 Skill 行为不变。
- 现有 Skill 页面上传接口继续创建 `shared` 资产。
- 会话自动打包创建 `private` 资产，并立即 materialize 为当前用户的 `UserSkill`。
- `list_skill_assets(shared=True)`、`find_shared_asset` 和 `ensure_shared_skills_for_user` 只处理 `shared`。
- 私有资产的读取、删除和采用操作必须验证 uploader，其他用户不可通过猜测 Skill ID 访问。

## 前端交互

生成进行中时，右下角圆形发送按钮切换为截图所示的圆形暂停按钮，使用方形停止图标：

- 点击后进入“正在暂停…”状态，按钮禁用防止重复请求。
- 取消接口成功后立即结束加载动画，在消息列表显示服务端的“已暂停本次生成”。
- 若原对话请求稍后返回，只接受 `cancelled` 结果，不追加回答。
- 新对话、切换历史会话和组件卸载时不会误取消其他会话的运行。
- 失败时恢复可点击的暂停按钮并显示明确错误；仍保留当前运行 ID，允许重试取消。

自动打包成功时：

- 助手消息明确显示“已生成、已上传、已启用、仅自己可见”。
- 顶部元信息增加 `Skill · 名称` 标签。
- 打开锤子 Skill 选择器时能立即看到新 Skill，并可用 `@skill-id` 调用。
- 打包失败时不创建半成品资产，回答给出具体失败点；不得回退成“请手动创建文件并上传”。

## API 契约

### `POST /api/agent/chat/`

请求新增：

```json
{
  "run_id": "uuid",
  "message": "把这次对话打包成 Skill 并自动上传",
  "conversation_id": "uuid"
}
```

普通成功沿用现有响应。取消响应新增：

```json
{
  "ok": false,
  "cancelled": true,
  "run_id": "uuid",
  "conversation_id": "uuid"
}
```

Skill 生成成功响应新增：

```json
{
  "ok": true,
  "reply": "已生成并启用……",
  "created_skill": {
    "skill_id": "workflow-name-session8",
    "name": "Workflow Name",
    "description": "...",
    "visibility": "private",
    "enabled": true
  }
}
```

### `POST /api/agent/runs/{run_id}/cancel/`

- 仅运行所有者可调用。
- 对 `running` 转为 `cancelled` 并创建占位消息。
- 对已 `cancelled` 返回同一结果，不重复创建消息。
- 对 `completed` 返回 409，避免把已完成回答伪装成暂停。
- 未找到或不属于当前用户统一返回 404，避免泄露运行存在性。

## 错误处理与一致性

- `run_id` 无效或重复属于其他请求时返回 400/409。
- 会话打包必须在数据库事务中完成资产记录与个人启用关系；任一步失败都不暴露半成品。
- COS 未配置时遵循现有 repository 的本地存储路径并保持 `private`；COS 已配置但上传失败时终止生成并回滚数据库记录，不伪装成上传成功。
- LLM 返回非 JSON、空字段、非法名称或疑似凭据时拒绝上传并给出可重试错误。
- 生成服务不执行会话里出现的命令，只把经过校验的流程写入 Skill。
- 打包指令本身作为用户消息保存；生成成功或失败作为对应助手消息保存，保证刷新历史后状态一致。

## 测试策略

后端测试覆盖：

- 运行创建、所有权、幂等取消、完成后不可取消、占位消息唯一。
- `run_chat` 在各关键阶段响应取消，取消后不保存正式回答。
- 流式 LLM 解析、取消时关闭连接、非流式兼容路径。
- Skill 脚本取消会终止进程组。
- 打包意图正反例、完整历史读取、打包指令排除、凭据脱敏。
- 合法 zip 结构、Skill parser 验证、稳定 Skill ID、重复生成更新同一资产。
- 私有资产不可出现在共享列表或自动同步给其他用户；现有共享上传行为不回归。
- LLM 失败或校验失败时不创建资产。

前端测试覆盖：

- 发送时创建并传递 `run_id`。
- 加载时发送按钮变为暂停按钮。
- 点击暂停只调用一次取消接口并显示“正在暂停”。
- 取消响应不追加普通助手回答。
- Skill 生成结果更新标签与可调用状态。

验证包括 Django 全量测试、前端单元测试、TypeScript/Vite 构建，以及在浏览器中实际验证发送、暂停、刷新历史、再次发言和自动生成 Skill 的完整流程。

## 非目标与后续演进

- 本期不提供部分文本保留或“继续生成剩余内容”。
- 本期不增加 Celery、Redis 或跨进程任务恢复。
- 本期不自动把会话 Skill 发布给全团队。
- 本期不让模型任意生成并立即执行新脚本。
- 多实例部署时，可把 `ChatRun` 取消状态迁移到 Redis/pub-sub，并把同步 POST 演进为队列任务 + SSE；API 中的 `run_id` 和状态模型保持兼容。
