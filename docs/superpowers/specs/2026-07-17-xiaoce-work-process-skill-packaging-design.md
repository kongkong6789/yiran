# 小策bot工作过程播报与会话 Skill 自动打包设计

## 背景

当前正式 AI 对话入口位于 `/collab`，用户通过通讯录中的“小策bot”完成知识问答。现有 `main` 分支会在后台生成回答，但前端只显示等待状态；用户无法判断系统正在检索、调用 Skill、运行工具还是组织回答。

仓库中的 `codex/xiaoce-pause-theme-skill` 分支已经实现了小策运行记录、真实暂停、会话 Skill 提炼、私有资产隔离和自动启用，但这些能力被 `c6d69cc` 从 `main` 回退。该分支还包含全站黑白主题等与本需求无关的改动，不能整体恢复。

本次采用选择性恢复：复用已验证的小策运行、暂停和私有 Skill 打包实现，再增加真实、持久化、可折叠的工作过程播报。

## 目标

1. 小策bot生成回答时，实时显示经过精简的工作过程。
2. 工作步骤必须来自真实后端执行阶段，不使用前端定时器模拟。
3. 回答完成后自动折叠为“查看处理过程（N步）”，刷新或重新打开会话后仍可查看。
4. 暂停时停在当前步骤，失败时显示经过安全映射的具体失败环节。
5. 不展示提示词、密钥、原始工具参数、原始工具结果、SQL、异常堆栈或模型内部推理。
6. 用户明确要求把当前会话打包成 Skill 并上传时，系统自动提炼、校验、保存为私有 Skill，并为当前用户启用。
7. 第一版仅作用于小策bot单聊，不改变群聊 `@AI`。

## 不在本次范围

- 不恢复或修改全站黑白主题。
- 不将过程播报扩展到普通同事私聊或群聊 `@AI`。
- 不公开模型完整思维链。
- 不把原始会话全文、敏感凭据或模型凭空生成的脚本写入 Skill 包。
- 不把会话生成的 Skill 自动发布到团队共享仓库。
- 不提供从暂停位置继续生成。
- 不改变现有协作会话、风险分析和附件协议的产品行为。

## 方案选择

采用“选择性恢复已有能力并新增持久化进度事件”。

不整体恢复 `codex/xiaoce-pause-theme-skill`，避免把主题和监控面板改动带回 `main`。不从零重写运行与 Skill 打包，避免重复实现已验证的取消竞态、权限隔离、脱敏、ZIP 校验和事务边界。

## 总体架构

每次小策bot问答由一个持久化 `XiaoceRun` 表示。运行记录保存状态、当前阶段、公开步骤列表、失败原因和关联消息。后端通过统一的进度上报器在真实执行边界更新运行记录，并使用现有 WebSocket 推送完整运行快照；前端以 `run_id` 幂等替换本地状态。

生成期间，前端在消息流末尾显示自动展开的过程卡。完成、暂停或失败时，后端把最终步骤快照写入对应小策消息的 `meta`；历史接口与实时接口返回同一元数据，所以刷新后无需依赖浏览器内存恢复。

## 数据模型

### XiaoceRun

恢复独立的小策运行模型，并增加：

- `status`：`running`、`cancelled`、`completed` 或 `failed`；
- `current_stage`：当前公开阶段代码；
- `progress_steps`：有序的公开步骤 JSON 数组；
- `error_code` 与 `error_message`：经过安全映射的失败信息；
- 房间、发起用户、触发消息、取消消息和完成消息关联；
- 创建、取消、完成和更新时间。

每个公开步骤包含固定字段：

```json
{
  "code": "knowledge_search",
  "label": "已检索知识库与 PostgreSQL 数据",
  "status": "completed",
  "tool_count": 0,
  "detail": "",
  "started_at": "2026-07-17T12:00:00Z",
  "finished_at": "2026-07-17T12:00:01Z"
}
```

`label` 和可选 `detail` 必须来自服务端白名单或安全格式化器，模型不得直接控制其内容。

### CollabMessage.meta

恢复协作消息内部元数据，保存：

- `run_id`；
- `process_steps` 最终快照；
- `process_status`；
- `cancelled`；
- `created_skill` 的安全摘要。

普通消息的 `meta` 默认为空，不改变现有接口行为。

### SkillAsset.visibility

恢复 `private` 与 `shared` 可见性。现有资产迁移为 `shared`；会话生成的 Skill 始终为 `private`，只能由上传者读取、更新、删除和启用。共享列表、共享查找和全员自动同步只处理 `shared`。

## 后端组件

### XiaoceProgressReporter

新增聚焦的小策进度上报器，提供开始、完成、失败、暂停和更新工具数量的接口。每次更新都：

1. 校验阶段代码是否在白名单中；
2. 在事务中锁定当前运行；
3. 按步骤代码追加或更新同一条步骤，避免重复；
4. 保存当前阶段和时间；
5. 推送最新运行快照。

阶段白名单覆盖普通问答与打包流程中的真实动作，包括：理解问题、检索知识或数据、调用 Skill、运行工具、校验指标、组织回答、读取会话、脱敏、提炼 Skill、校验 Skill 包、上传并启用。

只在实际发生对应动作时创建步骤。例如没有调用 Skill 时不显示“正在调用 Skill”；工具数量来自实际执行结果，而不是模型文字推断。

### 小策执行服务

恢复 `XiaoceRun` 创建、幂等取消、完成与失败逻辑。发送小策消息时，前端生成 UUID 并提交 `run_id`；后端拒绝同一用户在同一小策房间内的第二个活动运行。

普通问答继续调用现有 `run_chat`，但向 RAG、PostgreSQL/知识检索、Skill、MCP、图片和 LLM 阶段传入进度回调与取消检查。长耗时调用在可行边界检查取消；LLM 流式读取和 Skill 子进程沿用已有协作式取消实现。

完成、暂停与失败通过数据库行锁决定唯一终态：

- 完成时保存正式回答和最终过程快照；
- 暂停时不保存正式回答，只保存唯一暂停消息及步骤快照；
- 失败时保存安全失败消息及步骤快照；
- 已进入终态的运行不能被另一条路径覆盖。

### 会话 Skill 打包服务

只有消息同时表达“当前对话/聊天记录”“Skill/技能”以及“打包/生成/上传”等明确语义时才触发。普通 Skill 咨询仍走正常问答。

打包流程：

1. 按时间读取当前小策bot会话的用户与助手消息；
2. 排除本次指令、系统消息、撤回/删除消息、暂停占位和失败消息；
3. 校验至少存在一轮完整问答；
4. 在送入模型前替换 API Key、Bearer Token、Cookie、密码和常见密钥格式；
5. 限制总字符数，优先保留首个任务目标和最近关键结果；
6. 要求模型返回严格 JSON：名称、描述、命令式说明和工作流摘要；
7. 生成只包含 `SKILL.md` 与 `references/workflow-summary.md` 的 ZIP；
8. 使用现有 Skill parser 校验目录、frontmatter 和必要章节；
9. 在同一事务中保存私有资产并为当前用户启用。

Skill ID 由会话首个任务型问题与房间短 ID 稳定生成。同一房间重复打包更新同一资产，不产生重复 Skill。

## API 与实时协议

小策发送接口接受 `run_id` 并返回 `active_run`。房间详情返回当前用户在该小策房间的活动运行，使刷新或切换房间后能恢复生成状态。

新增取消接口，仅允许当前登录用户取消自己在指定房间发起的活动运行。重复取消返回相同结果；已完成或已失败运行返回冲突；不存在或不属于当前用户的运行统一返回未找到。

WebSocket 事件增加 `xiaoce_runs` 列表，每项包含运行 ID、状态、当前阶段和完整公开步骤。前端收到同一 `run_id` 时覆盖旧快照。断线时使用现有房间轮询恢复。

## 前端交互

- 仅当前房间为小策bot单聊时启用本功能。
- 生成中在消息流末尾显示展开的过程卡；当前步骤为加载态，已完成步骤为完成态。
- 工具步骤只显示安全名称和数量，例如“已运行 2 个工具”。
- 最终回答到达后，过程卡自动折叠为“查看处理过程（N步）”，用户可再次展开。
- 暂停后保留已完成步骤，并把当前步骤标为“已暂停”。
- 失败后把当前步骤标为失败，并展示服务端安全错误文案。
- 历史小策消息从 `meta.process_steps` 恢复折叠卡。
- 打包成功消息展示 Skill 名称、ID、“仅自己可见”和“已自动上传并启用”，并刷新 Skill 选择器。
- 打包失败可重试，不提示用户手工制作或点击上传。

## 安全与隐私

- 过程播报是工作摘要，不是完整思维链。
- 阶段名称和错误详情由服务端白名单控制。
- 不把提示词、历史原文、SQL、工具参数、工具返回正文、密钥、Cookie、令牌、文件路径或堆栈写入步骤。
- Skill 生成前和生成后均执行敏感信息替换。
- 原始会话不进入 ZIP，模型不得凭空增加 `scripts/`。
- 私有资产不出现在共享列表，也不会自动同步给其他用户。

## 一致性与失败处理

- 同一用户在同一小策房间只允许一个活动运行。
- 进度更新、完成、暂停和失败均锁定主运行记录，防止竞态。
- WebSocket 推送失败不回滚数据库状态；轮询可恢复最新快照。
- Skill 保存与个人启用原子完成，失败或取消不留下半成品。
- 取消发生在资产提交前时立即终止；进入最终提交事务后由运行锁决定唯一结果。
- 未分类异常统一映射为“当前步骤执行失败，请稍后重试”，详细堆栈只进入服务端日志。

## 测试

后端测试覆盖：

- 进度步骤追加、同代码更新、顺序、工具数量和白名单；
- 刷新恢复、最终消息快照和 WebSocket 负载；
- 活动运行冲突、幂等暂停、失败和完成竞态；
- 取消后不保存正式回答；
- 普通问答各真实阶段的条件上报；
- 打包意图正反例、历史筛选、长度限制和凭据脱敏；
- ZIP 结构、稳定 Skill ID、重复更新、私有隔离和事务回滚；
- 打包取消时不生成资产。

前端测试覆盖：

- 仅小策bot显示过程卡与暂停按钮；
- 运行中展开、完成后折叠和手动展开；
- 暂停、失败与刷新恢复；
- 同一 `run_id` 的实时快照幂等更新；
- 打包成功摘要与 Skill 列表刷新；
- 普通协作会话和群聊 `@AI` 无回归。

最终验证运行相关 Django 测试、前端单测、前端构建和 `manage.py check`，再在浏览器手工验证小策bot普通回答、暂停、失败、刷新恢复、首次打包和重复打包。

## 验收标准

- 小策bot生成期间实时显示真实、脱敏的工作步骤。
- 回答完成后显示可展开的“查看处理过程（N步）”，刷新后仍存在。
- 暂停和失败停留在正确步骤，且不会泄露内部信息。
- 明确打包指令生成私有、自动上传并启用的 Skill。
- 同一会话重复打包更新原 Skill；失败和暂停不留下半成品。
- 第一版不改变群聊 `@AI`，也不恢复全站主题改动。
