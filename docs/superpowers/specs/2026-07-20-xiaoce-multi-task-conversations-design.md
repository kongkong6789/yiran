# 小策bot 多任务会话设计

## 背景

正式小策bot 对话入口位于 `/collab`。现有后端把小策bot 当作单聊联系人，创建时会复用同一对参与者的最近房间；前端又会把所有单聊的显示名覆盖为对端联系人名。因此，用户无法创建多个小策任务，也无法在列表中看到自定义任务名。

现有系统已具备可复用的 `CollabRoom`、房间标题 PATCH、房间 DELETE、每房间独立 `XiaoceRun`、房间级 `session_key`、草稿缓存和房间列表轮询。本功能在这些能力上扩展，不建第二套聊天系统。

## 目标

1. 用户可以创建任意多个小策bot 任务会话。
2. 每个任务拥有独立的名称、消息、附件、草稿、运行记录和对话上下文。
3. 不同小策任务可同时运行，切换会话不暂停后台任务。
4. 用户可手工重命名和永久删除任务会话。
5. 删除正在运行的任务时，同时取消该运行。
6. 保留普通单聊、群聊、小策回答、暂停、工作过程和会话 Skill 打包的现有行为。

## 非目标

- 不新建任务数据库模型，不做模型迁移。
- 不把小策任务改造成顶部多标签或独立页面。
- 不让小策bot 加入群聊。
- 不改变“同一用户在同一房间只能有一个活动运行”的约束。
- 不将未发送草稿做服务端或浏览器长期持久化；草稿只需在当前页面的会话切换期间保留。
- 不做跨房间联合上下文、任务合并或批量删除。

## 方案选择

已对比三种信息架构：

1. 小策任务和人员聊天混合在现有消息列表：改动小，但任务多时难以扫视。
2. 现有侧边栏内设置独立的“小策bot 任务”分组：任务集中，且不破坏现有页面结构。
3. 主对话区顶部使用多标签：并行感强，但任务多时拥挤，移动端表现较差。

选择方案 2。它在多任务可见性、现有协作页一致性和实现风险之间取得最好平衡。

## 信息架构与交互

### 侧边栏分组

“消息”页签下拆成两个分组：

- **小策bot 任务**：放置所有当前用户与小策bot 的单聊房间，分组标题右侧显示“＋”。
- **其他对话**：放置普通单聊和群聊，保持当前行为。

分组只改呈现，不改服务端排序。每组内继续使用 `updated_at` 的现有顺序。旧小策bot 房间自动进入小策任务分组，不迁移、复制或修改消息。

### 创建任务

点击“＋”立即创建并打开新房间，默认名称为“小策bot（新任务）”。成功后新任务插入小策分组并成为当前会话；失败时保留当前会话和输入内容，不留空房间。

从通讯录或 `?bot=xiaoce` 深链打开小策bot 时，继续打开最近使用的已有小策任务；完全没有任务时才创建首个房间。只有小策分组上的“＋”保证创建新房间。

### 任务项和运行状态

任务项主标题使用房间自定义标题，不再被“小策bot”联系人名覆盖。存在 `active_xiaoce_run.status === "running"` 时显示带文字的“处理中”标识；没有活动运行时继续显示最近消息摘要、时间和未读数。

当前房间通过 WebSocket 实时更新；后台房间通过现有 15 秒列表刷新更新状态。用户切回房间时，详情接口与 WebSocket 恢复完整进度。

### 重命名

小策任务提供两个入口：列表项的“…”菜单和主会话标题旁的编辑按钮。两者共用现有标题 PATCH 能力和同一重命名弹窗。

- 保存前去除首尾空格。
- 空名称不允许保存，服务端最多保存 120 个字符。
- 任务名不要求唯一，房间 ID 是唯一标识。
- 接口成功后才更新列表和页头，失败时保留原名称。
- 普通群聊的“修改群名”文案保留。共用后端校验文案改为“会话名称不能为空”。
- 小策任务重命名不生成群公告消息。

### 删除

列表项的“…”菜单和会话设置菜单都提供“删除任务”。删除必须二次确认：

- 普通状态说明将永久删除任务和全部聊天记录。
- 运行状态额外说明正在处理的任务也会停止。

服务端先在事务中将该房间的活动 `XiaoceRun` 转为取消态，再删除房间及级联数据。这条删除路径不创建“已暂停”消息，因为该消息会随房间立即删除。取消检查将“运行记录已被删除”视为取消；完成、失败和进度上报路径在房间已删除时安静结束，不写入消息或推送结果。不支持立即中断的外部调用在下一个协作式检查点停止，但已删除任务不得再产生用户可见回答。

删除成功后立即从列表、房间缓存和草稿缓存中移除该 ID。删除当前房间后，选择小策分组中下一个房间；分组为空时显示创建引导。删除失败时保留列表项和当前会话。

## 技术设计

### 数据边界与并行

每个小策任务是一个 `CollabRoom(room_kind="dm")`，参与者为当前用户和小策bot。多任务只意味着这对参与者可拥有多个房间。

房间 ID 分别隔离 `CollabMessage`、`XiaoceRun`、`session_key="collab:room:<room_id>"`、会话 Skill 读取历史和前端房间缓存。现有条件唯一约束 `uniq_running_xiaoce_per_user_room` 保持不变：它阻止同房间重复运行，但允许同一用户在不同房间并行运行。

### API

新增 `POST /api/collab/xiaoce-tasks/`，请求体可选包含 `title`。

- 需要登录，无标题时使用“小策bot（新任务）”。
- 始终新建房间，不调用 `_find_open_dm`。
- 确保小策bot 账号存在，创建两名参与者和独立欢迎消息。
- 房间、参与者和欢迎消息在同一数据库事务中创建，失败时全部回滚。
- 成功返回 HTTP 201 和完整 `CollabRoom` 载荷。

现有端点保留并扩展：

- `GET /api/collab/rooms/`：小策单聊的 `display_title` 使用 `room.title`，普通单聊继续使用对端显示名。
- `PATCH /api/collab/rooms/<room_id>/`：继续更新 `title`，空值返回 HTTP 400 和中性错误文案。
- `DELETE /api/collab/rooms/<room_id>/`：保持现有权限，增加小策活动运行的安全取消。
- `POST /api/collab/rooms/`：普通单聊行为不变；小策bot 请求仍返回最近使用的已有房间，无房间时创建首个。

前端客户端新增 `createXiaoceTask(body?: { title?: string }): Promise<CollabRoom>`，更新与删除继续使用 `updateCollabRoom` 和 `deleteCollabRoom`。

### 前端组件边界

新增聚焦的小策任务列表组件，避免继续向已较大的 `CollabRisk.tsx` 堆叠分组、菜单和呈现细节。组件职责是：

- 接收小策房间、当前房间 ID 和用户权限。
- 呈现分组标题、“＋”、处理状态、未读数和项菜单。
- 通过回调发出选择、创建、重命名和删除意图。
- 不自行拥有房间数据或发送 API 请求。

`CollabRisk.tsx` 保持房间数据和异步动作的唯一所有者，使用现有 `isXiaoceRoom` 拆分小策任务和其他对话。重命名弹窗改为同时支持群聊与小策任务的中性流程。

### 并行数据流

1. 用户在任务 A 发送消息，后端为房间 A 创建 `XiaoceRun A` 并立即返回。
2. 用户切到任务 B，前端把任务 A 的草稿、附件和视图状态保存在房间 A 缓存键下。
3. 用户在任务 B 发送消息，后端为房间 B 创建 `XiaoceRun B`。因房间 ID 不同，A 和 B 可同时为 `running`。
4. 当前房间通过 WebSocket 实时更新，非当前房间的列表状态通过现有 15 秒轮询刷新。
5. 用户返回任务 A 时，前端立即恢复本地缓存，并拉取房间详情校准消息和活动运行。

## 权限、安全与可访问性

- 小策任务仍是仅当前用户和 bot 参与的单聊。
- 普通用户只能列出、打开、重命名和删除自己参与的房间。
- 管理员现有旁观和删除权限保持不变，旁观仍为只读。
- 房间标题只作显示，不参与授权或运行关联；删除始终使用精确 UUID。
- “＋”、“…”、编辑和删除按钮使用明确 `aria-label`。
- 处理状态同时使用文字，不只依赖颜色。
- 列表项主按钮和菜单按钮分开，可分别键盘聚焦。
- 重命名弹窗打开后选中当前名称，回车保存，Escape 取消。
- 删除确认框明确说明永久删除和停止运行的后果。

## 错误处理

- 新建、重命名和删除具有独立加载态，防止重复提交。
- 创建失败时显示服务端错误并保留当前上下文。
- 重命名空值在前端拦截，后端作最终校验。
- 删除失败不先行移除列表项，避免还原闪烁和缓存不一致。
- 现有房间请求序号与 `activeIdRef` 校验保留，迟到请求不得覆盖新房间。
- 任务 A 失败、暂停或删除不修改任务 B 的状态或输入缓存。

## 测试策略

### 后端测试

1. 专用端点连续创建两个小策任务，断言房间 ID 不同、参与者正确、默认名正确且每房间各有欢迎消息。
2. 传入自定义名称，断言去除首尾空格并按 120 字符上限保存。
3. 对同一普通联系人两次调用现有单聊创建端点，断言仍复用同一房间。
4. 对已有小策房间调用通讯录创建流程，断言返回最近使用的房间。
5. 房间列表对小策任务返回房间标题作为 `display_title`，对普通单聊仍返回对端名。
6. 小策标题 PATCH 成功，空标题返回 HTTP 400，且不生成群公告。
7. 同一用户在房间 A 和 B 可同时创建 `running` 的 `XiaoceRun`，同一房间的第二个运行仍被拒绝。
8. 删除无活动运行的任务，断言房间、消息和运行记录级联删除。
9. 删除有活动运行的任务，断言后台完成与失败路径安静结束，不再创建消息或发布 WebSocket 事件。
10. 非参与者不能读取、重命名或删除其他用户的小策任务。

### 前端测试

1. 分组函数正确分离任意数量的小策任务和其他对话。
2. 任务项使用房间标题，有活动运行时显示“处理中”。
3. 点击“＋”调用专用创建 API，成功后打开返回房间，失败时不切换。
4. 任务项和页头均能打开重命名，空值被拦截，成功后两处同步更新。
5. 运行中和非运行任务显示不同删除文案，成功后移除正确 UUID 和对应缓存。
6. 在任务 A 输入未发送草稿后切到任务 B，再切回 A 时恢复草稿、附件和回复目标。

### 综合验证

- 运行相关 Django 测试、整个 `apps.collab` 测试集与 `manage.py check`。
- 运行前端单测和 `npm run build`。
- 浏览器验证：创建两个任务，任务 A 处理时启动任务 B，切回查看进度，刷新恢复，重命名，删除已完成和运行中任务。
- 浏览器回归：普通单聊仍复用旧房间，群聊重命名、删除、未读和房间切换无回归。

## 验收标准

1. 用户可通过“小策bot 任务”分组的“＋”创建多个独立任务。
2. 新任务默认名为“小策bot（新任务）”，名称可手工修改。
3. 任务列表显示房间自定义名，不统一回退为“小策bot”。
4. 同一用户可在两个不同小策任务中同时拥有活动运行，消息、进度、草稿和上下文不串房间。
5. 切换任务不暂停后台运行，刷新后可恢复房间和服务端运行状态。
6. 删除运行中任务会先取消运行，再永久删除房间和聊天记录，后台不得在删除后发布回答。
7. 普通单聊、群聊和现有小策回答、暂停、工作过程与 Skill 打包行为保持正常。
