〇、一句话结论

结构层（L1/L2）已可演示闭环，动力学层（L3/L4）代码为零。

目前平台已跑通「本体图谱(含 AGE 直读) → 指标/异常数据底座 → 圆桌多 Agent 会诊(图谱注入+回写+三格式产物) → SOP 编排+闸机+审计」这条演示主线。 但方案 v2 的核心价值主张——五步法生成 Loop（动态关系识别→CLD→闭环检测→证据评分→人工确认）——在代码库中没有任何实现： 没有 CausalLink、没有 loop_detect、OntRelation 上没有 polarity/delay 字段。 同时，全部 7 个业务连接器为 Mock，RAG 为 4 条硬编码语料，所有 API 无鉴权。

对应方案的渐进采纳路径：当前处于 Level 0（纯 Ontology 建模）末段，距离第一期目标（Level 1 手工 CLD 试点 + 确认 1~2 条 Loop）还差五步法工具链与真实业务数据两块。

三、还没做的（对照方案逐项）

3.1 Loops 引擎 —— 方案核心，代码为零 完全缺失

全局扫描 loop_detect / loop_simulate / leverage_analyze / CausalLink / FeedbackLoop / polarity / delay / PDC，业务代码零命中（仅方案文档本身提及）。具体缺口：

五步法步骤需要的能力现状
① 动态关系识别OntologyRelation + 时序数据 → CausalLink 候选筛选Relation 只有 label 一个语义字段，无时序关联分析
② 符号因果图 CLDRelation 扩展 polarity(+/-)、delay(Δt)、function ƒ(x)；Entity→Stock 五类映射规则模型无任何因果元数据字段；无 Stock/Flow 概念
③ 闭环检测loop_detect 图遍历闭合路径 + R/B 判定无
④ 证据评分数据相关性 + 专家标注 + SOP 支撑的加权置信度无
⑤ 人工确认Loop 候选评审 UI + 企微审批 + 写入 Loop 库无 Loop 模型、无评审页面、企微未对接

第三期的 loop_simulate（模拟）、杠杆点分析、回路健康监控同样为零。24 个回路模式库目前只存在于方案文档，未进入任何数据结构。

3.2 连接器 —— 全部 Mock Mock/骨架

connectors/base.py 注册了 7 个连接器（金蝶、吉客云、企微智能表格、飞瓜/蝉妈妈、店铺后台、审批、内部），全部是 MockConnector，返回伪造 external_id，无任何真实 HTTP/SDK 调用。方案要求的腾讯文档、微盘、NAS、WorkBuddy 连接器连注册都没有。这意味着：

  • 感知层无真实数据：吉客云订单/库存、金蝶财务无法进入数据底座（当前指标全靠 seed 演示数据）
  • 执行层无真实动作：SOP「通过」闸机后的执行是假执行
  • 五步法①的「时序数据」原料实际不存在

3.3 RAG —— 关键词占位 Mock/骨架

rag/service.py 是 4 条硬编码语料 + 字符级关键词打分，并非真实 LightRAG 检索。圆桌会议注入的「制度/SOP 资料」实际来自这 4 条假语料。AGE 图谱虽然连通了 LightRAG 的 workspace，但文本检索通道没有打通。

四、产品视角：技术链路差在哪

4.1 数据流五层链路体检

按方案的「感知→理解→决策→执行→复盘」逐层看，绿色实线为已通，黄色为 Mock 顶着，红色虚线为断点：

感知层（数据进来）

吉客云 API→ 金蝶云 API→ seed 演示数据→ PG 分层数仓 dwd→dws→ads→ 指标/异常 API

断点：真实业务数据入口。没有它，五步法①的时序原料、证据评分④的数据相关性都无从谈起。这是整条链路的第一优先级缺口。

理解层（知识组织）

AGE 图谱直读+缓存→ 本体 L1/L2 CRUD+镜像→ RAG(4条硬编码)→ L3 回路层(Stock/CausalLink/Loop)

断点：OntRelation 停在纯结构层。要进 Level 2，最小改动是给 Relation 加 polarity/delay_days/evidence_score/is_causal_candidate 四个字段，新增 Loop 模型（成员链、R/B 类型、置信分、确认状态）。

决策层（会诊与判断）

圆桌多Agent会诊→ 图谱注入+回写→ LLM(网关不可用,Mock顶替)→ Loop候选评审(五步法⑤)

决策层是全平台完成度最高的一层，但缺 Loop 评审这个「专用会议类型」：把候选 Loop（极性/延迟/证据分）作为议题注入圆桌，确认后写入 Loop 库。

执行层（动作落地）

闸机5道校验→ 审批流(仅标记无闭环)→ 连接器(全Mock)→ 审计留痕

断点：need_approval 之后没有下文——需要审批单模型 + 企微审批回调 + 批后续跑。连接器需要至少 1 个真实对接（建议吉客云只读）打样。

复盘层（回路监控）

审计日志查询→ PDC循环→ 回路健康监控→ 杠杆点分析/模拟

整层属于第二、三期，当前不阻塞，但依赖 L3 回路层先建起来。

五、业务视角：需要注意什么

⚠️ 演示数据 ≠ 业务事实

当前 DataLake 页看到的 GMV/毛利率/异常预警全部来自 seed 演示数据。在吉客云真实接入前，不要用平台数字做任何经营判断，也不要向依然公司演示时暗示这是真数。

⚠️ Mock 模式下的会议产物

LLM 网关不可用时，Agent 发言与最终方案由 Mock 模板生成，观点是「像样但空心」的。试点会议（如品牌退款率会诊）务必先修复网关，否则人工确认环节（五步法⑤）在给假分析背书。

⚠️ 不要跳过人工确认

方案核心原则「OntologyRelation ≠ Loop、人工确认不可省略」。即使 loop_detect 上线后，统计关联误判因果（假阳性）依然是最大方法论风险；业务负责人评审极性/延迟这一步是刚性流程，不是可选项。

⚠️ 组织投入先于工具

第一期 Level 1 是「腾讯文档手工画 CLD + 圆桌验证」，不依赖新代码。需要提前锁定 2~3 名业务骨干 + 1 名 IT 对接人，安排 2~3 次各 2 小时的 CLD 工作坊。工具再好，没人标注极性/延迟就是空转。

⚠️ 软存量数据缺口

品牌声誉、代理能力指数等 TYPE IV 衍生存量没有直接数据源。第一期建议降级为「专家打分的定性 Loop」，不要为凑数据强行拼指标。

⚠️ 对外使用前必须加鉴权

所有 API 当前完全开放（AllowAny），角色由调用方自报。内网单人演示可接受，但接入依然公司真实订单/财务数据前，鉴权 + 角色服务端化是红线，否则闸机的权限/预算校验形同虚设。
